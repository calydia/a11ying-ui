# RichText Heading IDs

## Goal

Make heading fragment identifiers generated by `RichText` readable, stable,
ASCII-compatible, and unique across one rendered RichText document. Table of
contents links and rendered headings must always use the same identifiers.

## Current Problem

`RichText` currently slugifies headings independently during table of contents
collection and content rendering. The implementation removes non-ASCII
letters, so Finnish text can produce damaged or empty identifiers. Repeated
headings also receive the same identifier, including collisions between normal
headings and ContentBox headings.

Nested RichText content can introduce further duplicate identifiers because
each nested component calculates identifiers without knowledge of its parent
document.

## Behavior

Heading identifiers are derived in document order.

1. Convert heading text to lowercase.
2. Normalize it with Unicode NFKD decomposition.
3. Remove combining marks so accented Latin letters are transliterated to
   their ASCII base letters.
4. Replace remaining non-ASCII letters, punctuation, symbols, and whitespace
   runs with a single hyphen.
5. Remove leading and trailing hyphens.
6. Use `heading` for a normal heading with no usable slug and `content-box` for
   a ContentBox heading with no usable slug.
7. Allocate duplicate suffixes in document order: `overview`, `overview-1`,
   `overview-2`.

For example:

- `Näkö ja ääni` becomes `nako-ja-aani`.
- `Hello, world!` becomes `hello-world`.
- Three `Overview` headings become `overview`, `overview-1`, and
  `overview-2`.

Uniqueness applies across all headings rendered by one top-level `RichText`,
including normal headings, ContentBox headings, and headings inside nested
ContentBox or DisclosureWidget content.

## Architecture

`RichText` will create one deterministic heading plan for the complete
document tree before rendering. Each planned heading records its generated
identifier and whether it belongs in the table of contents.

Both content rendering and table of contents rendering will consume this same
plan. The content renderer will no longer slugify heading text independently.
Nested RichText content will receive the relevant shared plan or rendering
context instead of starting a separate identifier namespace.

The public `RichText` API remains unchanged. Any internal props or context
needed to share the plan stay private to the package.

## Table Of Contents Scope

This change preserves current table of contents membership:

- top-level normal headings are included;
- ContentBox headings encountered by the existing top-level traversal are
  included;
- headings inside DisclosureWidget content or a ContentBox body are not newly
  added to the table of contents.

Nested headings still participate in the page-wide identifier allocation, so
they cannot collide with headings that are listed in the table of contents.

## Error And Edge Handling

- Missing heading text uses the stable type-specific fallback.
- Repeated fallback identifiers receive the same numeric suffix treatment as
  ordinary slugs.
- Mixed inline formatting contributes its text in document order.
- Identifier generation is deterministic for the same RichText node tree.
- No CMS-provided identifier is introduced as part of this change.

## Testing

Focused unit tests will verify:

- Finnish accented letters transliterate to ASCII;
- punctuation and whitespace normalize predictably;
- duplicate normal headings receive ordered suffixes;
- normal and ContentBox headings share one collision namespace;
- empty or symbol-only headings use stable fallbacks;
- nested headings participate in collision allocation without changing table
  of contents membership;
- every table of contents fragment points to the exact identifier rendered on
  its target heading.

The RichText Storybook table of contents fixture will include Finnish accented
text and duplicate headings. Existing Storybook accessibility and visual tests
will verify the rendered component.

After the shared package is built and released, both `a11ying-front` and
`wcag-front` will update the dependency and lockfile. Each consumer will run
focused RichText/page checks followed by its required quality gate because the
change affects shared rendered content and navigation.

## Ownership

All identifier generation and collision handling belongs in `a11ying-ui`
because both sites use the shared RichText renderer. The consumers require no
local slugging implementation or API changes.
